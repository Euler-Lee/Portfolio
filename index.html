Estudos de Caso (Hands-on)

1. Tuning de Query: De Seq Scan para Index Scan Performance PostgreSQL
O Cenário (Diagnóstico)
Um relatório financeiro crítico estava levando minutos para ser gerado. Utilizei o EXPLAIN ANALYZE e identifiquei um Sequential Scan em uma tabela de 15 milhões de registros devido ao uso de funções na cláusula WHERE que invalidavam o índice existente.
A Solução e Resultado
Criei um índice funcional baseado na expressão exata da query.

Resultado: O custo da query caiu de 250.000 para 150. O tempo de execução baixou de 4 minutos para menos de 1 segundo. 
-- PLANO ANTES: Seq Scan on pagamentos (cost=0.00..254320.00)
SELECT * FROM pagamentos WHERE date_part('year', data_pag) = 2024;

-- SOLUÇÃO: Criação de Índice Funcional
CREATE INDEX idx_pagamentos_ano ON pagamentos (date_part('year', data_pag));

-- PLANO DEPOIS: Bitmap Heap Scan using idx_pagamentos_ano (cost=0.00..152.00)
2. Resolução de Deadlocks e Travamento de Aplicação Troubleshooting
Identificação via CloudWatch
Alerta de "Database Connections" e "CPU Utilization" disparou no CloudWatch simultaneamente. A aplicação parou de responder. Acessei o banco e consultei a view pg_stat_activity filtrando por transações com estado 'active' há muito tempo.
Ação Imediata e Correção
Identifiquei dois processos competindo pelos mesmos recursos em ordem inversa. Realizei o pg_terminate_backend no processo bloqueador para liberar a fila.

Posteriormente, orientei a equipe de dev a padronizar a ordem de update das tabelas para evitar recorrência. 
-- Query utilizada para identificar quem bloqueia quem
SELECT pid, usename, state, query 
FROM pg_stat_activity 
WHERE wait_event_type = 'Lock' 
AND state != 'idle';
3. Incidente Crítico Oracle RDS (Storage Full) AWS & Oracle
O Alerta
Recebi notificação crítica do AWS CloudWatch: FreeStorageSpace abaixo do limite seguro em uma instância Oracle RDS de produção. O auto-extend do volume físico da AWS havia atingido o limite configurado.
Ação de Contenção
Conectei via client, verifiquei as tablespaces e confirmei que a tablespace de DADOS estava cheia. Adicionei um novo Datafile imediatamente utilizando as packages proprietárias do RDS (rdsadmin), normalizando o serviço em menos de 5 minutos, sem downtime. 
-- Verificação e Adição de Datafile no ambiente RDS
SELECT tablespace_name, used_percent FROM dba_tablespace_usage_metrics;

-- Comando executado para expansão imediata
ALTER TABLESPACE TSD_DADOS ADD DATAFILE SIZE 5G AUTOEXTEND ON NEXT 1G;
