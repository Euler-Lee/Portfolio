1. PostgreSQL: Refresh Físico Paralelo de 4,2 TB (de 12h para 2h) Automação & Engenharia PostgreSQL
Contexto e Escala
Base Protheus em PostgreSQL 16 de 4,2 TB. O refresh de produção para homologação levava cerca de 12 horas com métodos sequenciais (dump/restore e basebackup single-stream), obrigava janela noturna e ainda disputava I/O com a produção, atrasando os times de desenvolvimento a cada ciclo.
Diagnóstico
Instrumentei cada etapa e descartei o storage: o gargalo não era o volume lógico nem o disco, e sim CPU single-thread. A cópia sequencial saturava um único núcleo enquanto os demais ficavam ociosos. Ler direto da produção ainda somava risco de contenção de I/O no ambiente crítico.
Solução
Reescrevi o processo como clone físico paralelo lendo de uma réplica (não da produção): cópia low-level dos diretórios de dados distribuída em 16 processos, pg_receivewal fixando o ponto de consistência e reescrita de roles (tpprd para tphml) no destino. Reconstruí o storage de homologação em RAID0 de 4 vias para vazão e migrei o backup para a réplica (retenção por contagem). O endurecimento do script revelou e corrigiu 4 bugs (watch de progresso, streaming de WAL, tratamento do código 24 do rsync e troca de role no clone físico).
Resultado e Validação
Janela caiu de ~12h para ~2h, executável em horário comercial e sem impacto na produção. Processo comprovado em execução real, com a homologação em uso pelo cliente e o backup diário na réplica validado por ciclos consecutivos. 
# 1. Streaming de WAL da réplica para fixar o ponto de consistência
pg_receivewal -h replica -D /wal_stage -S refresh_slot &

# 2. Cópia física paralela dos diretórios de dados (16 processos, lendo da réplica)
ls -d /replica/pgdata/base/*/ \
  | xargs -P16 -I{} rsync -a --whole-file {} /hml/pgdata/base/

# 3. Reescrita de role no destino (produção -> homologação) e start
psql -h hml -c "REASSIGN OWNED BY tpprd TO tphml;"
pg_ctl -D /hml/pgdata start
2. Oracle: ORA-00018 por Saturação de Shared Server Troubleshooting Oracle
Contexto e Escala
Instância Oracle 10g de um ERP legado (aplicação de coletores em campo) passou a rejeitar novas conexões com ORA-00018 (maximum number of sessions exceeded), interrompendo a operação em produção.
Diagnóstico
A instância rodava em modo Shared Server com apenas 1 dispatcher. Cruzando v$dispatcher, v$shared_server, v$resource_limit e v$session, confirmei que o pool de sessões de shared server estava no teto: a aplicação abria conexões em rajada, cada uma consumindo uma sessão, e o único dispatcher não dava vazão.
Solução
Mitigação imediata com bounce controlado da instância para liberar as sessões travadas e restabelecer o atendimento. Como correção estrutural, especifiquei ampliar dispatchers e shared servers, mover as conexões persistentes de integração para modo dedicado e revisar o pool da aplicação para conter as rajadas.
Resultado e Validação
Serviço normalizado logo após o bounce, com as conexões voltando a ser aceitas. Plano de correção definitiva documentado para eliminar a recorrência, aguardando janela de aprovação do cliente. 
-- Confirmação da saturação do shared server
SELECT name, status, round(busy/(busy+idle)*100,1) AS pct_busy FROM v$dispatcher;
SELECT resource_name, current_utilization, max_utilization, limit_value
FROM v$resource_limit
WHERE resource_name IN ('sessions', 'processes');

-- Correção estrutural proposta
ALTER SYSTEM SET DISPATCHERS = '(PROTOCOL=TCP)(DISPATCHERS=3)';
ALTER SYSTEM SET SHARED_SERVERS = 10;
3. SQL Server: Recuperação de Instância com Domínio Indisponível Recovery / Disaster
Contexto e Escala
Instância SQL Server 2016 de produção não iniciava após a indisponibilidade do controlador de domínio. Toda a operação do ERP do cliente estava parada, com pressão máxima por retorno.
Diagnóstico
O serviço falhava no start com erro de logon: a conta de serviço era de domínio e, sem o DC no ar, não autenticava. O Configuration Manager e o log de erro confirmavam parada por falha de credencial, não por corrupção de banco, o que direcionou a estratégia de recuperação.
Solução
Reconfigurei a conta de serviço para uma conta local do sistema (LocalSystem), removendo a dependência do domínio, e reiniciei a instância. Com o banco no ar, validei integridade física e lógica com DBCC CHECKDB e conferi recovery model e estado de cada base.
Resultado e Validação
Banco disponível em minutos, sem perda de dados e CHECKDB limpo. Registrei a pendência de retorno a uma conta de serviço gerenciada após a normalização do domínio e sinalizei o risco de EOL da versão para planejamento de upgrade. 
-- Com a instância no ar, validação de integridade
DBCC CHECKDB (N'ERP_PROD') WITH NO_INFOMSGS, ALL_ERRORMSGS;

-- Conferência do estado e recovery model de cada base
SELECT name, state_desc, recovery_model_desc
FROM sys.databases
ORDER BY name;
4. SAP HANA: Falha Diária de Backup por OOM do Indexserver SAP HANA
Contexto e Escala
Backup diário de um tenant SAP HANA (Business One) falhava de forma recorrente, deixando o ambiente sem ponto de recuperação confiável por vários dias, um risco crítico de negócio.
Diagnóstico
Duas causas combinadas. O indexserver era encerrado por OOM durante o backup (memória insuficiente para a carga somada ao backup simultâneo) e a chave hdbuserstore da rotina apontava para a porta errada do SystemDB (30015 em vez de 30013), fazendo a automação falhar já na autenticação.
Solução
Ajustei o global_allocation_limit para 48 GB e ampliei o swap para 9 GB, eliminando o OOM, e recriei a chave do hdbuserstore na porta correta do SystemDB (30013).
Resultado e Validação
Backup diário validado com restore real do tenant, restabelecendo o ponto de recuperação e encerrando a necessidade de intervenção manual noturna. 
-- Ajuste do limite de alocação de memória (global.ini): 48 GB
ALTER SYSTEM ALTER CONFIGURATION ('global.ini', 'SYSTEM')
  SET ('memorymanager', 'global_allocation_limit') = '49152' WITH RECONFIGURE;

# Correção da chave hdbuserstore para a porta correta do SystemDB (30013)
hdbuserstore SET SYSTEMDB "host:30013" SYSTEM <senha>

# Validação: backup real do tenant e verificação no catálogo
hdbsql -U SYSTEMDB "BACKUP DATA FOR <TENANT> USING FILE ('backup_val')"
