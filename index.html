Estudos de Caso (Casos Reais)

1. Oracle: ORA-00018 e Saturação de Shared Server Troubleshooting Oracle
O Cenário (Diagnóstico)
Um ERP de produção começou a rejeitar novas conexões com o erro ORA-00018 (maximum number of sessions exceeded). A instância operava em modo Shared Server com apenas um dispatcher, e o crescimento de conexões da aplicação saturou os shared servers e o pool de sessões. Diagnostiquei pelas views v$dispatcher, v$shared_server e v$resource_limit.
A Solução e Resultado
Ampliei o número de dispatchers e de shared servers e ajustei os limites de sessions e processes para a carga real, aliviando a fila imediatamente. Para os serviços de integração com conexões persistentes, recomendei conexão dedicada.

Resultado: erro eliminado e conexões normalizadas, sem downtime de aplicação. 
-- Diagnóstico da saturação de shared server
SELECT name, status, round(busy/(busy+idle)*100,1) AS pct_busy FROM v$dispatcher;
SELECT resource_name, current_utilization, limit_value
FROM v$resource_limit
WHERE resource_name IN ('sessions', 'processes', 'shared_server_sessions');

-- Alívio: mais dispatchers e shared servers
ALTER SYSTEM SET DISPATCHERS = '(PROTOCOL=TCP)(DISPATCHERS=3)';
ALTER SYSTEM SET SHARED_SERVERS = 10;
2. PostgreSQL: Refresh Físico de Produção para Homologação Automação PostgreSQL
O Cenário (Diagnóstico)
A cópia de produção para homologação (base multiterabyte) levava muitas horas e exigia janela noturna, atrasando os times de desenvolvimento. Ao instrumentar o processo, identifiquei que o gargalo real era CPU single-thread dos métodos sequenciais, não o disco.
A Solução e Resultado
Construí um clone físico paralelo lendo de uma réplica: cópia low-level dos diretórios de dados em 16 vias, pg_receivewal para garantir o ponto de consistência, e troca de role de produção para homologação no destino.

Resultado: janela reduzida para cerca de 2 horas, executável em horário comercial, com o backup movido para a réplica para não impactar a produção. 
# WAL contínuo da réplica para o ponto de consistência
pg_receivewal -h replica -D /wal_stage -S refresh_slot &

# Cópia paralela dos diretórios de dados (16 processos)
ls -d $PGDATA/base/*/ | xargs -P16 -I{} rsync -a {} hml:/pgdata/base/

# Ajuste de role no destino: produção -> homologação
psql -h hml -c "ALTER DATABASE app OWNER TO app_hml;"
3. SQL Server: Recuperação de Instância Crítica Fora do Domínio Recovery
O Cenário (Diagnóstico)
Uma instância SQL Server de produção não subia após a queda do controlador de domínio: o serviço falhava ao iniciar porque a conta de serviço era de domínio e não autenticava. O ERP do cliente estava totalmente parado. O erro no log apontava falha de logon da conta de serviço.
A Solução e Resultado
Reconfigurei o serviço do SQL Server para uma conta local (LocalSystem), subindo a instância imediatamente e de forma independente do domínio, e validei a integridade com DBCC CHECKDB.

Resultado: banco disponível sem perda de dados, com plano posterior de retorno à conta de serviço adequada após a normalização do domínio. 
-- Após subir a instância, validação de integridade
DBCC CHECKDB (N'ERP_PROD') WITH NO_INFOMSGS, ALL_ERRORMSGS;

-- Conferência do estado dos bancos
SELECT name, state_desc, recovery_model_desc
FROM sys.databases
ORDER BY name;
4. SAP HANA: Falha Diária de Backup por Pressão de Memória SAP HANA
O Cenário (Diagnóstico)
O backup diário de um tenant SAP HANA falhava de forma recorrente. O diagnóstico apontou o indexserver sendo encerrado por OOM (falta de memória) durante o backup, somado a uma chave hdbuserstore do SystemDB apontando para a porta errada.
A Solução e Resultado
Ajustei o global_allocation_limit e ampliei o swap para eliminar o OOM, e corrigi a chave do hdbuserstore para a porta correta do SystemDB.

Resultado: backup diário validado com restore real do tenant, encerrando a necessidade de intervenção manual. 
-- Ajuste do limite de alocação de memória (global.ini)
ALTER SYSTEM ALTER CONFIGURATION ('global.ini', 'SYSTEM')
  SET ('memorymanager', 'global_allocation_limit') = '49152' WITH RECONFIGURE;

# Correção da chave hdbuserstore do SystemDB (porta correta)
hdbuserstore SET SYSTEMDB "host:30013" SYSTEM <senha>
